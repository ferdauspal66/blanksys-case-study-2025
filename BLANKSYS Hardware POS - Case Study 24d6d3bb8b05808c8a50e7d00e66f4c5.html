Design Process

1. Research & Persona

UI UX Designer (Firdaus) melalukan research dengan 3 metode yaitu

  • User interview: Melakukan tanya jawab langsung dengan client yang akan menggunakan system ini yaitu pemilik toko bangunan
  • Competitor research: Melakukan review dan analisis kepada aplikasi sejenis untuk mengetahui kelebihan dan kekurangan yang mana ditemukan ada 2 aplikasi yang menjadi acuan referensi
    • Toqoo (Point of sales)
    • Manager.io
  • Brainstorming with AI: Melakukan diskusi dan brainstorming mendalam dengan AI seperti chatgpt, gemini dan claude untuk mengetahui alur bisnis yang efektif untuk client.

2. User Flow & System Concept

Setelah mendapatkan konsep yang fixed dari 3 metode tadi maka ditentukan konsep awal dari alur system nya dengan melakukan beberapa tahapan berikut

  1. Membuat user journey map dengan figjam
  1. Pembuatan list modul system
  1. Pembuatan alur logika dan kebijakan dari system

3. Wireframe

Di langkah berikutnya kami membuat design kasar dari system yang sederhana. Dari sini sudah kelihatan wujud dari systemnya seperti apa nantinya secara konsep, layout dan alur penginputan.

4. User Interface Design

Dalam tahap ini, fokus utama adalah membentuk identitas visual dari BLANKSYS Hardware POS agar selaras dengan kebutuhan fungsional dan preferensi estetika client.

  • Design System: Shadcn UI
  • Font: Raleway (sesuai permintaan client, dipilih karena bentuknya modern, bersih, dan mudah dibaca)
    Hierarki Font :
    • Heading (H1 – H3): Bold, ukuran besar untuk penekanan fitur utama.
    • Subheading (H4 – H6): Medium, digunakan pada label form dan subfitur.
    • Body Text: Regular, ukuran sedang agar nyaman dibaca pada layar POS maupun perangkat mobile.
  • Color Pallete
  • Komponen Utama (Mengikuti shadcn Design System)
    • Buttons: Primary (hitam 800, teks putih), Secondary (hitam 600, teks putih), Outline (border 100).
    • Form Elements: Input dengan border radius 8px, shadow subtle, dan state fokus berwarna hitam 600.
    • Tables: Header bold, row dengan striping, dan hover highlight menggunakan warna hitam 700.
    • Cards: Digunakan untuk menampilkan ringkasan data di dashboard, dengan shadow soft dan rounded 8-12px.

5. Prototyping

  • Linking Pages: Semua halaman dashboard, inventory, sales, dan purchase dihubungkan dengan navigasi interaktif.
  • Interactive Components:
    • Dropdown untuk pemilihan kategori barang.
    • Modal pop-up untuk tambah data barang dan supplier.
    • Transition animasi ringan untuk berpindah antar halaman.
  • Device Simulation: Prototype diuji dalam mode desktop POS (monitor) dan tab sesuai dengan perangkat kasir di toko namun tetap disediakan responsive mobile untuk monitoring mudah dari owner
  • User Scenario Testing dalam Figma:
    • Simulasi pembelian
    • Simulasi penjualan
    • Simulasi akunting

6. User Testing

Tahap ini memastikan bahwa desain yang dibuat benar-benar sesuai dengan kebutuhan pengguna sebelum masuk tahap full development.

Metode yang digunakan:

  • Moderated Usability Testing (tatap muka & via Google Meet).
  • Participants:
    • 2 Pemilik toko bangunan (target utama).
    • 2 Kasir toko (pengguna operasional).
    • 1 Admin stok gudang.

Flow Pengujian:

  1. Task Scenario:
    • Menambahkan barang baru ke inventory.
    • Melakukan transaksi penjualan.
    • Membuat laporan penjualan harian.
  1. Metrics yang diukur:
    • Task Completion Rate (berapa banyak tugas yang berhasil diselesaikan).
    • Time on Task (waktu yang dibutuhkan per tugas).
    • Error Rate (kesalahan navigasi / input).
  1. Feedback Collection:
    • Form evaluasi setelah pengujian.
    • Catatan verbal saat proses testing.

Development Process

TechStack Overview

Environment Setup

Kami menggunakan VPS IDCloudhost dengan OS Ubuntu sebagai server utama. Untuk memudahkan pengelolaan, server dikonfigurasi menggunakan custom admin console berlangganan, sehingga monitoring resource, deployment, dan update sistem dapat dilakukan lebih cepat dan terkontrol.

Tech Stack & Rationale

Frontend

ReactJS + TypeScript (SPA)

  • Alasan Pemilihan:
    • Single Page Application (SPA) memastikan navigasi antar halaman sangat cepat karena hanya memuat data baru, bukan reload seluruh halaman.
    • TypeScript meningkatkan keamanan kode dengan type-checking, meminimalisir bug saat pengembangan fitur kompleks seperti transaksi, laporan, dan manajemen stok.
    • React memiliki ekosistem komponen yang luas dan fleksibel, memudahkan integrasi dengan design system shadcn.
  • Implementasi:
    • Component-based architecture untuk modularitas.
    • Optimasi rendering dengan React.memo dan lazy loading pada halaman berat seperti laporan penjualan.

Backend

Node.js + ExpressJS + NestJS

  • Alasan Pemilihan:
    • Node.js unggul untuk aplikasi real-time dan scalable.
    • ExpressJS digunakan untuk API routing ringan.
    • NestJS memberikan struktur proyek yang rapi, dependency injection, dan maintainability jangka panjang.
    • Kombinasi ini memberi fleksibilitas untuk membuat API REST maupun GraphQL.
  • Implementasi:
    • API modular berdasarkan domain (inventory, sales, purchases, accounting).
    • Service Layer di NestJS mengatur business logic agar mudah di-maintain.

Database

MySQL untuk data transaksional.

PostgreSQL untuk laporan analitik dan data agregasi.

  • Alasan Pemilihan:
    • MySQL cepat dan ringan untuk operasi CRUD harian.
    • PostgreSQL unggul dalam query kompleks dan agregasi data.
  • Implementasi:
    • Replikasi data dari MySQL ke PostgreSQL menggunakan job scheduler.
    • Indexing di tabel transaksi untuk mempercepat query laporan.

Cloud Server

Self-Hosting di VPS IDCloudhost

  • Alasan Pemilihan:
    • Kontrol penuh terhadap konfigurasi server.
    • Hemat biaya dibandingkan layanan cloud publik untuk skala awal.
    • Mendukung deployment kustom sesuai kebutuhan arsitektur monorepo.

Security

  • JWT Access Token & Refresh Token
    • Autentikasi aman dan efisien.
    • Pisah masa aktif access token (pendek) & refresh token (panjang) untuk keamanan ekstra.
  • RBAC (Role-Based Access Control)
    • Batasi akses fitur berdasarkan peran (Owner, Kasir, Admin Gudang).
    • Minimalkan risiko kesalahan data akibat akses yang tidak relevan.

Project Architecture

Monorepo

  • Alasan Pemilihan:
    • Semua frontend, backend, dan library internal dalam satu repositori.
    • Sinkronisasi antar tim developer lebih mudah.
    • Mengurangi overhead dependency management.
  • Implementasi:
    • Menggunakan pnpm workspace untuk manajemen paket.
    • Shared types antar frontend & backend untuk menghindari mismatch struktur data.

3. Deployment Process

  1. Build Process
    • Frontend: build dengan Vite (cepat & bundle kecil).
    • Backend: build dengan Nest CLI.
  1. Server Configuration
    • Reverse proxy dengan Nginx.
    • SSL/TLS diaktifkan.
  1. Continuous Deployment
    • GitHub Actions untuk otomatisasi build & deploy.
  1. Monitoring
    • PM2 menjaga proses Node.js tetap aktif.
    • Log terintegrasi ke custom admin console.
